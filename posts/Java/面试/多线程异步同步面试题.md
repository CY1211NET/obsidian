# 1️⃣ 同步 vs 异步

## ✔ 同步（Synchronous）

- 当前任务未完成时，线程**阻塞等待**
    
- 任务串行执行
    
- 典型代码：普通方法调用、MVC 单线程处理流程
    

**适用场景：**

- 任务间强依赖（登录 → 获取权限 → 加载主页）
    
- CPU 密集型逻辑
    
- 简单业务流程
    

---

## ✔ 异步（Asynchronous）

- 当前任务未完成，线程**不阻塞**
    
- 任务可并行
    
- Java 中常用 CompletableFuture、线程池来实现
    

**适用场景：**

- I/O 密集型（数据库、RPC、网络调用）
    
- 多个服务可并发调用
    
- 提升吞吐量、减少等待时间
    
- 响应式架构、微服务聚合

# 2️⃣ Java 线程创建方式（4 种）

1. **继承 Thread**（不推荐，扩展性差）
    
2. **实现 Runnable**（推荐）
    
3. **实现 Callable + FutureTask**（有返回值）
    
4. **使用线程池 ThreadPoolExecutor**（企业中使用最多）
    

---

# 3️⃣ 为什么需要线程池？

### ✔ 避免频繁创建/销毁线程（昂贵）

每次 new Thread 都要创建栈、PCB，成本高。

### ✔ 控制最大并发，避免系统崩溃

无限创建线程会 OOM。

### ✔ 能复用线程，提高吞吐量

### ✔ 提供队列、拒绝策略、线程管理机制

真正适合高并发、大流量服务。

---

# 4️⃣ ThreadPoolExecutor 核心参数（非常重要）

|参数|含义|
|---|---|
|corePoolSize|核心线程数|
|maximumPoolSize|最大线程数|
|keepAliveTime|非核心线程空闲多久回收|
|workQueue|调度队列|
|threadFactory|创建线程方式|
|rejectedExecutionHandler|拒绝策略|

---

# 5️⃣ 线程池种类（Executors）

|线程池|特点|是否推荐|
|---|---|---|
|newFixedThreadPool|固定线程数量，队列无限|❌ 可能 OOM|
|newCachedThreadPool|线程数几乎无限|❌ 容易爆线程|
|newSingleThreadExecutor|单线程顺序执行|✔ 特定场景|
|newScheduledThreadPool|定时任务|✔ 定时场景|
|自定义 ThreadPoolExecutor|参数可控|✔ 企业推荐|

# 6️⃣ 线程池状态（五种）

线程池有五类运行状态：

1. **RUNNING**：接受任务 & 执行任务
    
2. **SHUTDOWN**：不接受新任务 + 执行队列任务
    
3. **STOP**：中断所有线程 + 不执行队列
    
4. **TIDYING**：全部任务完成、线程回收中
    
5. **TERMINATED**：彻底关闭
    

状态流转图：

```
           shutdown()
 RUNNING ───────────→ SHUTDOWN ────→ TIDYING → TERMINATED
    │                     ▲
    │ shutdownNow()       │ 
    └────────────→ STOP ──┘
```

# 7️⃣ 线程池常见问题与分析

## ✔（1）死锁

情况：

- 线程池线程数量不足
    
- 线程 A 等待线程池执行任务 B
    
- B 无法获得线程 → 死锁
    

解决：

- 避免在池内等待池内任务
    
- 放大线程池
    
- 使用 CompletableFuture 链式写法
    

---

## ✔（2）队列满导致 RejectedExecutionException

原因：

- workQueue 满
    
- thread >= maximumPoolSize
    
- 触发拒绝策略
    

解决：

- 增大 maximumPoolSize
    
- 增大队列
    
- 选择 CallerRunsPolicy
    
- 限流、降级
    

---

## ✔（3）线程泄漏

原因：

- 任务永不结束
    
- 忘记关闭线程池
    
- 死循环
    
- 阻塞 IO 永远不返回
    

影响：

- 线程越积越多 → OOM → 服务挂掉
    

解决：

- 给任务设置超时
    
- 任务可中断
    
- 合理 shutdown
    
- 监控线程池运行情况

# 8️⃣ 实际开发：前端并发 vs 后端聚合 + 线程池

### ☑ 做法 A：前端并发调用多个接口（不用线程池）

优点：

- 后端简单
    
- 接口粒度清晰
    
- 前端灵活（要哪个调哪个）
    

缺点：

- 前端必须维护多个请求
    
- 网络开销大
    
- 接口鉴权/降级分散
    
- 不适合复杂场景
    

---

### ☑ 做法 B：后端聚合接口 + 线程池并发查询

优点：

- 前端只调一次
    
- 网络开销低
    
- 后端统一做认证/容错/降级/超时
    
- 后端可用 CompletableFuture 并发查询，提高性能
    

缺点：

- 聚合层逻辑复杂（拼装数据）
    
- 接口复用性较低
    
- 容易变“大而全”接口
    
- 需要合理设计线程池

# 🔟 综合总结（面试可直接背）

```
同步是阻塞的、串行的，适用于逻辑简单且任务有依赖的场景；
异步是非阻塞并行执行的，适用于 I/O 密集和多服务并发调用。

线程的创建方式包括 Thread、Runnable、Callable、线程池，
实际开发基本都用 ThreadPoolExecutor。

线程池通过线程复用、队列和拒绝策略提升吞吐量并防止系统被打爆。
线程池有五种状态：RUNNING、SHUTDOWN、STOP、TIDYING、TERMINATED。

常见问题包括死锁、队列满、线程泄漏，需合理设置 core/max、队列大小、
拒绝策略和任务超时。

对于聚合业务，前端并发适合简单情况；
复杂微服务场景建议后端用线程池并发调用多个下游并统一治理。
```
