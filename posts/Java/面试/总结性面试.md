# **1️⃣ MyBatis 一级缓存 & 二级缓存区别？你项目有没有开？为什么？**

### ✅ **标准回答：**

**MyBatis 一级缓存（本地缓存）**

- **作用范围：SqlSession 级别**
    
- 同一个 SqlSession 内，相同查询（相同 SQL + 参数）会直接走缓存，不再查数据库。
    
- 默认开启，不需要额外配置。
    
- SqlSession 关闭缓存就失效，或者执行增删改也会清空缓存。
    

**MyBatis 二级缓存（全局缓存）**

- **Mapper 映射级别**（不同 SqlSession 共享）
    
- 需要手动开启（XML 或 @CacheNamespace）
    
- 底层依赖序列化，把结果缓存到磁盘或内存中
    

### ❗ 为什么我在项目里没有开二级缓存？

（这个回答很好，适合面试）

> **在我的实际项目中，我没有使用 MyBatis 二级缓存。主要原因是：**  
> 1）我的项目查询数据更新较频繁（例如论文状态更新、评审进度、查重结果）  
> 2）二级缓存会带来**缓存不一致的问题**，需要额外管理失效  
> 3）我们采用 Redis 做缓存，性能更高，分布式场景更适合  
> 4）系统是多节点部署，MyBatis 二级缓存无法跨节点

所以我只保留一级缓存，不使用二级缓存。

# **2️⃣ 事务隔离级别？InnoDB 默认是什么？为什么？**

### 🧠 **四个隔离级别从低到高：**

1. **READ UNCOMMITTED（读未提交）**  
    会出现脏读
    
2. **READ COMMITTED（读已提交）**  
    会出现不可重复读
    
3. **REPEATABLE READ（可重复读）**  
    MySQL InnoDB 默认  
    可避免不可重复读，但可能出现幻读
    
4. **SERIALIZABLE（串行化）**  
    性能最差，完全避免并发问题
    

---

### 🔥 **InnoDB 的默认级别：REPEATABLE READ**

好背的回答：

> MySQL InnoDB 默认是 **可重复读（RR）**，因为它通过 **MVCC（多版本并发控制）+ Next-Key Lock** 可以在性能和一致性之间取得平衡。
> 
> 在 RR 下可以避免脏读和不可重复读，而且 InnoDB 的 Next-Key Lock 机制也能避免幻读。

# **3️⃣ Spring Boot 自动装配原理？（面试常问）**

# **5️⃣ Redis 如何实现分布式锁？风险？如何避免？**

### 🎯 **获取锁**：

```shell
SET lock_key value NX PX 5000
```

说明：

- **NX**：只有 key 不存在才创建（防重复加锁）
    
- **PX 5000**：自动过期，避免死锁
    

---

### 🎯 **释放锁（不能直接 DEL）**

原因：不同客户端可能错误删除别人的锁

正确做法：用 Lua 脚本保证原子性：

```lua
if redis.call("get",KEYS[1]) == ARGV[1] then
    return redis.call("del",KEYS[1])
else
    return 0
end
```

---

### ❗ 分布式锁存在的风险：

1. **锁过期但业务还没完成 → 竞争问题**
    
2. **时钟漂移 → 过期时间失效**
    
3. **Redis 主从同步延迟（主挂了后锁丢失）**
    

---

### 🔥 **如何避免？**

- 使用 **合理的过期时间**
    
- 业务分阶段检查锁
    
- 使用 **Redisson 的分布式锁**（推荐）
    
- 使用看门狗机制自动续期（Redisson 提供）

# ✅ **1. JVM 的内存结构包含哪些区域？各自的作用是什么？**

## ⭐ 标准回答（简短版）

JVM 内存主要分为：

1. **程序计数器（PC 寄存器）**：记录当前线程执行的字节码指令地址（线程私有）。
    
2. **虚拟机栈（JVM Stack）**：保存方法调用时的栈帧，包括局部变量、操作数栈等（线程私有）。
    
3. **本地方法栈（Native Method Stack）**：为本地方法（Native）服务（线程私有）。
    
4. **堆（Heap）**：存放对象实例，是 GC 的主要区域，分为新生代 + 老年代（线程共享）。
    
5. **方法区（Method Area / 元空间 Metaspace）**：存放类元数据、常量池、静态变量等（线程共享）。
    
6. **运行时常量池（Runtime Constant Pool）**：方法区的一部分，存放各种常量信息。
    

## ⭐ 加分解释

- Java8 之后方法区由 **永久代（PermGen）替换为元空间（Metaspace）**，元空间使用直接内存，不容易发生 OOM PermGen Error。
    
- 堆是 GC 主要处理对象的区域，分为 **年轻代（Eden、S0、S1）** 和 **老年代**。

# ✅ **2. JVM 的 GC 回收机制？年轻代和老年代怎么回收？**

## ⭐ 标准回答（简短版）

GC 分为两个部分：

### **年轻代：使用“复制算法”**

新对象 → Eden  
Eden 满 → Minor GC  
存活对象 → S0/S1  
反复复制，存活对象少 → 复制算法快。

### **老年代：使用“标记-整理（Mark-Compact）”**

当对象晋升或年轻代多次存活 → 进入老年代。  
老年代对象存活率高，所以使用 **标记-整理**，避免碎片。

## ⭐ 加分回答

常见 GC 垃圾收集器：

- **年轻代：Serial / ParNew / G1 Young**
    
- **老年代：CMS / G1 Old**
    

Minor GC → 很快  
Major / Full GC → 慢、会 Stop-The-World

# ✅ **3. final、finally、finalize 的区别？**

## ⭐ 标准回答（简短版）

- **final**：关键字
    
    - 修饰变量：不可变
        
    - 修饰方法：不可被重写
        
    - 修饰类：不可被继承
        
- **finally**：异常处理中的块
    
    - 无论是否异常都会执行，用于释放资源
        
- **finalize()**：Object 方法
    
    - GC 回收前调用（不可靠、不推荐）
        
    - Java9 已被废弃
        

## ⭐ 面试加分点

Java 以后不会依赖 finalize，因为不可控，会影响 GC 性能。

# ✅ **4. HashMap 的底层实现？扩容机制？线程安全吗？**

## ⭐ 标准回答（简短版）

**底层结构**：

- Java8 之后：**数组 + 链表 + 红黑树**
    
- 链表长度 > 8 → 转红黑树；树节点小于 6 → 退回链表
    

**扩容机制**：

- 当 size 达到 **负载因子（0.75） × 容量** 时触发扩容
    
- 扩容时数组容量变为原来的 2 倍
    
- 节点重新 hash 到新数组（rehash）
    

**是否线程安全？**

- HashMap **不是**线程安全的，线程不安全体现在：
    
    - 并发 put 会导致数据覆盖
        
    - 扩容时链表可能形成**环**（死循环）
        

## ⭐ 面试加分点

线程安全替代：

- **ConcurrentHashMap**（分段锁 → CAS + synchronized）

# ✅ **5. 什么情况下会发生 ConcurrentModificationException？怎么避免？**

## ⭐ 标准回答（简短版）

发生 CME 的典型场景：

1. **遍历集合的同时修改集合（非安全迭代器）**  
    例如：
    

```java
for (Integer i : list) {
    list.remove(i); // CME
}
```

2. 底层依赖 **fail-fast** 机制，modCount 发生变化即抛异常。
    

---

## ⭐ 怎么避免？（面试非常爱问）

✔ 使用 **Iterator 的 remove()** 方法

```java
Iterator<Integer> it = list.iterator();
while (it.hasNext()) {
    if (it.next() == 1) it.remove();
}
```

✔ 使用线程安全集合

- CopyOnWriteArrayList
    
- ConcurrentHashMap
    

✔ 使用 **for 循环倒序遍历删除**

```java
for (int i = list.size() - 1; i >= 0; i--) {
    list.remove(i);
}
```

✔ 使用 Java8 stream 过滤后生成新集合：

```java
list = list.stream().filter(x -> x != 1).toList();
```

# ✅ 1️⃣ 线程的生命周期有哪些阶段？

## ⭐ 标准回答（可直接说）

Java 线程有 6 个生命周期状态：

1. **New（新建）**：创建 Thread 对象但未 start。
    
2. **Runnable（就绪/运行）**：调用 start() 后，只要获得 CPU 时间片就能运行。
    
3. **Blocked（阻塞）**：等待获取 synchronized 的锁。
    
4. **Waiting（无限等待）**：调用 wait()、join()、park()，等待其他线程唤醒。
    
5. **Timed Waiting（限时等待）**：sleep(n)、wait(n)、join(n)。
    
6. **Terminated（终止）**：线程执行完毕或异常退出。
    

## ⭐ 加分点（面试官喜欢）

Blocked 和 Waiting 的区别：

- Blocked 是“等待锁”
    
- Waiting 是“等待条件/通知”

# ✅ 2️⃣ synchronized 和 ReentrantLock 的区别？

## ⭐ 标准回答

**1）语法不同：**

- synchronized：关键字，自动加锁 + 自动释放锁
    
- ReentrantLock：类，需要手动 lock()、unlock()
    

**2）功能差异：**  
ReentrantLock 更强大：

- 可实现**公平锁**
    
- 可中断锁（lockInterruptibly）
    
- 可超时获取锁（tryLock）
    
- 支持多个 Condition 条件队列
    
- 支持监控锁是否被持有
    

**synchronized 功能少但更简单、性能稳定**

**3）底层区别：**

- synchronized 基于 **JVM 级别**（Monitor 对象）
    
- ReentrantLock 基于 **Java 代码**（AQS，AbstractQueuedSynchronizer）
    

## ⭐ 面试加分总结（背这句就赢了）

> synchronized 简单、自动释放；  
> ReentrantLock 功能更强，可中断、可公平、可定时、可绑定多个条件队列。

# ✅ 3️⃣ volatile 能保证原子性吗？为什么？

## ⭐ 标准回答（非常简短）

**volatile 不能保证原子性，只保证可见性与禁止指令重排。**

例如：

```java
volatile int count = 0;
count++;
```

仍然不是原子操作，因为它实际上是：

1. 读取 count
    
2. +1
    
3. 写回 count  
    多线程会发生竞态条件。
    

## ⭐ volatile 能保证什么？

- **可见性**：线程修改变量后立即对其他线程可见。
    
- **有序性**：禁止指令重排序（保证 happens-before）。
    

## ⭐ 加分点：什么时候用 volatile？

- 状态标记（stop flag）
    
- 配合双重检查锁（DCL）使用
    
- 不涉及复合操作的共享变量
    

> 简单一句：**volatile 保证可见性，不保证原子性。**

# ✅ 4️⃣ ThreadLocal 的使用场景是什么？有什么风险？

## ⭐ 使用场景（面试必答）

ThreadLocal 是为**每个线程**创建独立变量副本，实现 **线程隔离**。

常见场景：

1. **保存用户登录信息（比如拦截器）**
    
2. **数据库连接、Session、Transaction 绑定到当前线程**
    
3. **格式化工具（SimpleDateFormat 非线程安全）**
    
4. **每个请求独立的上下文对象**
    

> 核心作用：**让线程有自己独立的变量，不会互相干扰。**

---

# ⚠️ ThreadLocal 的风险（面试官爱问）

最大风险：**内存泄漏**

为什么？

- ThreadLocalMap 的 key 是弱引用
    
- 但是 value 是强引用
    
- 线程池复用线程 → ThreadLocal 的 value 不会自动释放
    
- 可能一直占内存 → 内存泄漏
    

## ⭐ 如何避免？

- 使用完要 **手动 remove()**
    
- 不要在核心逻辑里无限设置 ThreadLocal
    
- 在线程池中尤其要清理 ThreadLocal
    

面试一句话总结：

> ThreadLocal 会导致内存泄漏，尤其在线程池中，一定要 remove()。

> 线程有 6 个状态：新建、就绪、运行、阻塞、等待、终止。

> ThreadLocal 为每个线程提供独立变量，用于保存线程上下文，但要注意内存泄漏，尤其在线程池要 remove()。

# ✅ 1️⃣ Spring 中 Bean 的生命周期是怎样的？

## ⭐ 标准回答（可以直接背）

Spring Bean 生命周期大致分为 7 个阶段：

1. **实例化（Instantiation）**：创建对象
    
2. **属性赋值（Populate Properties）**：依赖注入
    
3. **BeanNameAware / BeanFactoryAware 回调**
    
4. **初始化（Initialization）**
    
    - 调用 `@PostConstruct`
        
    - 调用 `InitializingBean.afterPropertiesSet()`
        
    - 调用 `<bean init-method="">`
        
5. **BeanPostProcessor 前置处理（before）**
    
6. **BeanPostProcessor 后置处理（after）**
    
7. **销毁（Destroy）**
    
    - 调用 `@PreDestroy`
        
    - 调用 `DisposableBean.destroy()`
        
    - 调用 `<bean destroy-method="">`
        

## ⭐ 加分一句话总结

> Bean 生命周期包含：实例化 → 依赖注入 → 初始化 → 使用 → 销毁。  
> Spring 使用 BeanPostProcessor 允许我们在初始化前后进行扩展。

# ✅ 2️⃣ AOP 的原理是什么？解决什么问题？

## ⭐ 标准回答（简洁）

AOP 本质是 **代理模式**。  
Spring 的 AOP 由两种代理实现：

1. **JDK 动态代理**（目标类有接口）
    
2. **CGLIB 动态代理**（没有接口，使用子类代理）
    

AOP 解决的问题是：

> 把日志、事务、安全认证、监控等 **横切关注点** 从业务代码中剥离出来，实现解耦，提高代码复用性。

## ⭐ 加分解释（可以多说一点）

AOP 的核心概念：

- **切面**（Aspect）
    
- **通知**（Before、After、Around…）
    
- **切点**（Pointcut）
    
- **连接点**（Join point）

# ✅ 3️⃣ @Transactional 底层是怎么实现事务的？哪些情况会失效？

## ⭐ 标准回答

@Transactional 通过 **AOP 代理 + 动态代理** 实现：

1. 方法调用入口 → 被代理对象拦截
    
2. Spring 创建数据库事务（Connection）
    
3. 执行方法逻辑
    
4. 正常返回：提交事务
    
5. 异常抛出：回滚事务
    

事务本质基于：

- **数据库事务机制（InnoDB）**
    
- **Spring 的事务管理器（PlatformTransactionManager）**
    

---

## ⚠️ @Transactional 失效的常见情况（面试高频）

### ❌ 1. 调用同类中的方法（自调用）

代理没生效。  
例：

```java
this.method(); // 事务不生效
```

### ❌ 2. 方法不是 public

Spring 事务只对 public 方法有效。

### ❌ 3. 异常被捕获，没有抛出

事务回滚不了。

### ❌ 4. 使用的事务传播行为不适配

比如 REQUIRES_NEW、NOT_SUPPORTED 等。

### ❌ 5. 数据库不支持事务（MyISAM）

---

## ⭐ 面试一句话总结

> @Transactional 底层基于 AOP 动态代理和数据库事务。事务会在自调用、非 public 方法、异常被捕获等情况下失效。

# 🚨 一、Spring **循环依赖**（高概率必问）

## ✅ 1. **什么是循环依赖？**

循环依赖是指：

> A 依赖 B，B 又依赖 A，Spring 在创建 Bean 的过程中出现相互依赖的情况。

典型例子：

```java
class A {
    @Autowired
    private B b;
}

class B {
    @Autowired
    private A a;
}
```

---

# ✅ 2. **Spring 如何解决循环依赖？（重点回答）**

Spring 使用“三层缓存”解决大部分循环依赖（构造器注入除外）：

---

## **🔷 三层缓存（非常关键）**

|缓存|名称|存什么？|作用|
|---|---|---|---|
|**1 级缓存**|singletonObjects|**完全初始化好的 Bean**|最终成品|
|**2 级缓存**|earlySingletonObjects|**提前曝光的 Bean（半成品）**|解决循环依赖|
|**3 级缓存**|singletonFactories|**创建 Bean 的工厂，能生成代理对象**|AOP 场景|

---

## ⭐ **Spring 解决循环依赖的完整流程（面试黄金答案）**

> Spring 创建 A → 发现依赖 B → 去创建 B  
> B 创建过程中又依赖 A → Spring 发现 A 正在创建 →  
> 从 **三级缓存** 拿到 A 的“早期引用（半成品）”注入到 B → B 完成初始化 → 回来完成 A → 最终放入一级缓存。

一句话总结：

> **Spring 用三级缓存提前曝光 Bean 的引用，使得互相依赖能顺利完成。**

---

# ❗ 3. 哪些情况 Spring **不能**解决循环依赖？

面试官经常追问：

### ❌ 1）**构造器注入**会导致循环依赖无法解决

```java
class A { A(B b){} }
class B { B(A a){} }
```

因为 A、B 都需要“先创建实例”，无法提前曝光。

### ❌ 2）@Scope("prototype") 原型模式不能被三级缓存缓存

没有缓存 → 无法提前曝光 → 循环依赖失败

### ❌ 3）Bean 被 AOP 代理但条件不满足，Spring 无法提前暴露

---

# ⭐ 面试加分总结（背这几句即可）

> Spring 通过 **三级缓存机制**解决大部分循环依赖。  
> 本质是**提前暴露未完全初始化的单例对象**，让另一个依赖方可以引用。  
> 构造函数注入 + 原型模式无法解决循环依赖。

## ✅ 1. 什么是 IOC？

# ✅ 2. BeanFactory 和 ApplicationContext 的区别？

这是面试常问：

## ⭐ **BeanFactory（底层容器）**

- Spring 最核心的容器
    
- **懒加载**（第一次用 Bean 时才创建）
    
- 只提供基本的依赖注入功能
    

适用于轻量级场景（很少直接用）

---

## ⭐ **ApplicationContext（高级容器）**

- BeanFactory 的子接口（功能超集）

    - 事件发布机制（ApplicationEvent）
        
    - AOP 自动代理
        
    - 自动装配、更完整的生命周期管理
        

使用场景：**实际开发都是用 ApplicationContext**

---

# 📌 简单总结一句话（面试金句）

> **BeanFactory 是 Spring 最核心的 IOC 容器，提供基本依赖注入。  
> ApplicationContext 是它的增强版，默认预加载单例 Bean，并支持 AOP、事件等高级功能。**

# 🔮 IOC 容器启动流程（加分）

# ❌ **IOC ≠ 反射（但会使用反射）**

# 🔥 **那 IOC 到底是什么？**

一句话：

> **IOC（控制反转）是把对象的创建与依赖管理从代码中拿出来，交给 Spring 容器统一创建和管理。**

也就是：

- 以前你写代码时自己 `new A()`
    
- 现在你是什么都不用管，由 Spring 创建 A 并注入依赖 B
    

这是“控制权反转”：  
对象的控制权从**开发者** → **Spring 容器**。

# 🔧 **那反射是干什么的？**

反射是 Java 的一种机制，用来在运行时：

- 创建对象（`newInstance()`）
    
- 访问属性
    
- 调用方法
    

Spring 在创建 Bean 时，需要动态创建对象、注入属性  
→ 所以用到反射来实现。

---

# ⭐ IOC 和反射的关系（可以在面试里直接说）

你可以用一句话总结：

> IOC 是一种设计思想，把对象的创建和管理交给容器；  
> 反射是 Java 提供的技术手段；  
> **Spring 的 IOC 是用反射来实现的，但两者不是同一个概念。**

面试官会非常满意。

# 🟦 **什么是 Java 反射机制？**

一句话：

> **反射是 Java 在运行时动态获取类的信息，并操作类、方法、属性的能力。**

平时 new 对象是“编译时确定类型”，  
反射是“运行时操作类型”。

---

# 🟩 **反射能做什么？（非常面试关键）**

反射允许程序在运行时：

1. **获取类的信息**
    
    - 类名、父类、接口、构造方法、属性、方法
        
2. **创建对象**
    
    ```java
    Class<?> clazz = Class.forName("com.demo.User");
    Object obj = clazz.newInstance();
    ```
    
3. **操作属性**
    
    ```java
    Field field = clazz.getDeclaredField("name");
    field.setAccessible(true);
    field.set(obj, "Tom");
    ```
    
4. **调用方法**
    
    ```java
    Method method = clazz.getMethod("sayHello");
    method.invoke(obj);
    ```
    

---

# 🟨 **为什么需要反射？（场景非常重要）**

你一定要记住这些常用场景（面试常问）：

### **① 框架底层依赖反射（Spring、MyBatis、Hibernate）**

- Spring 用反射创建 Bean、注入属性、调用方法
    
- MyBatis 通过反射将 ResultSet 填充到对象中
    
- AOP 动态代理底层也依赖反射
    

### **② 解耦代码，支持动态加载**

比如读取配置文件后动态加载类：

```properties
className=com.demo.MyServiceImpl
```

运行时加载：

```java
Class.forName(className).newInstance();
```

### **③ JSON 序列化、RPC 框架**

例如 FastJSON、Jackson 会通过反射解析对象属性。

---

# 🟧 **反射的三种使用方式（必须会）**

### **1. 通过类名字符串获取 Class 对象**

```java
Class<?> clazz = Class.forName("com.demo.User");
```

### **2. 通过对象获取**

```java
User user = new User();
Class<?> clazz = user.getClass();
```

### **3. 通过类.class 获取**

```java
Class<User> clazz = User.class;
```

> 这三种方法都能得到 Class 对象，再通过 Class 做所有事情。

---

# 🔴 **反射的缺点（面试高频）**

1. **性能较低**  
    因为需要检查类型、安全性，JVM 难以优化。  
    （但可以用 MethodHandle 或反射缓存提升）
    
2. **破坏封装性**  
    用反射可以修改 private 属性、调用 private 方法：
    
    ```java
    field.setAccessible(true);
    ```
    
3. **容易被滥用**  
    会让代码可读性变差，不推荐业务逻辑里用反射。
    

---

# 🟣 **反射与 IOC 关系（重要）**

你可以这样总结（面试官非常爱听）：

> IOC 是思想：对象和依赖交给容器管理；  
> 反射是技术：Spring 用它创建对象、依赖注入。  
> **IOC ≠ 反射，但 Spring IOC 是靠反射实现的。**

---

# 🟢 **可直接背诵的一句话总结**

> Java 反射是一种在运行时动态获取类信息并操作对象的机制，  
> 广泛应用于 Spring、MyBatis、序列化、动态代理等框架。  
> 虽然强大，但性能开销大，会破坏封装，因此一般用于框架底层，而非业务代码。

# ✅ **1. MySQL 的索引为什么使用 B+ 树？**

## ⭐ **标准回答：**

MySQL（InnoDB）使用 **B+ 树作为索引结构**，主要原因是：

### **1）磁盘访问次数最少（核心原因）**

- B+ 树的每个节点能存储很多 key（多路平衡树）
    
- 树的高度非常低（一般 2～4 层）
    
- 查找任何数据只需 **2～4 次磁盘 IO**
    

相比之下：

- **二叉树**太高 → 磁盘 IO 多
    
- **B 树**每个节点既存 key 又存 data → 可存储数据少，树会变高
    
- **哈希索引**无顺序、无法范围查询
    

### **2）B+ 树是“磁盘友好型”结构**

- 所有数据都存储在叶子节点
    
- 内节点仅存 key，不存数据 → 节点更小 → 一次 IO 可加载更多 key
    

### **3）支持范围查询和排序（B+ 树天然有序）**

- 比如 `BETWEEN`、`> <`、`ORDER BY` 都可高效响应
    
- 哈希索引无法做到
    

---

# 🟦 一句话总结：

> MySQL 使用 B+ 树是因为磁盘 IO 次数少、层级低、节点大、范围查询高效，是最适合数据库索引的结构。

---

# ✅ **2. InnoDB 与 MyISAM 的区别？**

|特性|InnoDB|MyISAM|
|---|---|---|
|**事务**|✔ 支持|❌ 不支持|
|**行锁**|✔ 行级锁|❌ 表级锁|
|**外键**|✔ 支持|❌ 不支持|
|**崩溃恢复**|✔ 有恢复能力|❌ 容易损坏|
|**计数操作**|较慢（行扫描）|快（存储了行数）|
|**适用场景**|高并发 OLTP|读多写少 OLAP|

## ⭐ 一句话总结：

> InnoDB 支持事务、行锁、崩溃恢复，更适合互联网高并发业务。  
> MyISAM 查询快，但不支持事务及行锁，现在几乎不用。

# ✅ **3. 事务的 ACID 特性是什么？InnoDB 如何实现？**

## ⭐ ACID 解释：

1. **Atomicity 原子性**  
    操作要么全成功，要么全失败
    
2. **Consistency 一致性**  
    数据必须从一个一致状态到另一个一致状态
    
3. **Isolation 隔离性**  
    多个事务互不影响
    
4. **Durability 持久性**  
    提交后的数据不会丢失
    

---

## ⭐ InnoDB 如何实现 ACID？

### **1）原子性：Undo Log（回滚日志）**

- 操作前记录一个 undo log
    
- 失败可以 Rollback
    

### **2）一致性：依赖其他三个特性一起实现**

### **3）隔离性：MVCC + 锁机制**

- 多版本控制（MVCC）实现快照读
    
- 行锁 + 间隙锁（Gap Lock）避免幻读
    

### **4）持久性：Redo Log（重做日志）**

- 提交时先写 redo log，再持久化到磁盘（WAL 机制）
    

---

# 🟧 一句话总结：

> InnoDB 通过 Undo Log、MVCC、锁、Redo Log 实现 ACID，是事务型数据库的核心。

# ✅ 1️⃣ MySQL 的锁有哪些？行锁、表锁、间隙锁？

MySQL（InnoDB）提供的锁分为三类：

---

## ⭐ **（一）按锁的粒度分类**

### **1. 表锁（Table Lock）**

- 锁整张表
    
- MyISAM 默认
    
- 并发性能差
    

使用场景：DDL（更改表结构）

---

### **2. 行锁（Row Lock）**🚀（InnoDB 默认）

- 锁某一行
    
- 并发高、粒度小
    
- 适用于高并发 OLTP 系统
    

例如：

```sql
select * from user where id = 10 for update;
```

---

## ⭐ **（二）按锁的类型分类（行锁内部）**

### **1. 记录锁（Record Lock）**

锁一条具体记录，比如 `id=10`

### **2. 间隙锁（Gap Lock）**

“锁某一段索引区间”，比如 (10, 20)

作用：

> **防止插入新数据，避免幻读**

### **3. Next-Key Lock（记录锁 + 间隙锁）**

锁“当前记录 + 前后间隙”，InnoDB RR 隔离级别默认使用

作用：

> 解决幻读问题

---

# 🔥 一句话总结（能打动面试官）

> MySQL 有表锁、行锁；行锁又包括记录锁、间隙锁、Next-Key Lock。  
> 间隙锁和 Next-Key Lock 的核心作用是防止幻读。

# ✅ 2️⃣ Explain 如何分析 SQL 性能？

你要重点关注 explain 的以下字段：

|字段|含义|面试要点|
|---|---|---|
|**type**|访问类型|最重要：ALL > index > range > ref > eq_ref > const|
|**possible_keys**|可用索引|有无可用索引|
|**key**|实际使用的索引|NULL 说明未使用索引|
|**rows**|预计扫描行数|越小越好|
|**Extra**|额外信息|是否出现 Using filesort、Using temporary|

---

## ⭐ 最重要：type（访问类型）

从差到好：

```
ALL（全表扫描）
index
range
ref
eq_ref
const / system（最好）
```

> type 越靠后越好。ALL 是最差，说明 SQL 没有用到索引。

---

## ⭐ Extra 字段（重点关注）

- **Using index** = 覆盖索引（不回表） → 最快
    
- **Using where** = 用到 where 条件过滤（正常）
    
- **Using filesort** ⚠ 排序使用外部文件（性能差）
    
- **Using temporary** ⚠ 使用临时表（group by 常见）
    

---

# 🔥 一句话总结（必背）

> Explain 中最重要的是 type、key、rows、extra。  
> 遇到 ALL、Using filesort、Using temporary 基本就是慢 SQL 了。

# ✅ 3️⃣ 覆盖索引、回表是什么？

## ⭐ 覆盖索引（Covering Index）

> 查询的数据 **完全来自于索引**，不用回到表里查。

例如：

```sql
select id, name from user where id = 10;
```

若索引是 (id, name)，那直接查索引就能得到数据。

**好处：**

- 无需回表
    
- 性能最高（减少一次磁盘 IO）
    

---

## ⭐ 回表（Back to Table）

> 使用索引找到主键值后，还要回到主键索引（聚簇索引）查数据。

例如：

```sql
select age from user where name = 'Tom';
```

如果 name 是单列索引，只能从 name 索引中拿到 **主键 ID**，然后回表。

**性能较差，但正常情况都会回表。**

---

# 🔥 一句话总结

> 覆盖索引避免回表，是优化 SQL 最常见手段。

---

# ✅ 4️⃣ MySQL 为什么要回收 Undo Log？

Undo Log 的作用：

- 实现 **原子性**（回滚）
    
- 实现 **MVCC 多版本控制**
    

当更新数据时，会生成 undo log 记录旧值。

### ❗ 但有问题：

Undo Log 是不断增长的：

- 如果没有清理，会无限膨胀，导致：
    
    - 磁盘占用暴增
        
    - 性能急剧下降
        
    - 数据库变慢甚至崩溃
        

因此 MySQL 必须进行 “undo 回收（purge）”。

---

## ⭐ Undo 回收机制（非常关键）

Undo Log 在满足以下条件时会被标记为可回收：

1. **对应的事务已经提交**
    
2. **没有其他事务需要查询它的历史版本（快照）**
    

即：

> 所有使用该 undo 的 ReadView 都已经失效 → 可清理。

MySQL 的 purge 线程会异步清理这部分 undo。

---

# 🔥 一句话总结

> Undo Log 是为了回滚和 MVCC，但会不断增长，因此需要 purge 线程异步清理。

---

# ✅ 5️⃣ MVCC 原理（ReadView 详细讲解）

## ⭐ MVCC 是什么？

MVCC = **多版本并发控制**

解决：

- 读写冲突
    
- 提高并发能力
    

核心机制：

1. Undo Log
    
2. ReadView（读视图）
    

---

# 🔥 ReadView 原理（最重要）

### 当一个事务执行普通 select（快照读）时，会创建 ReadView。

ReadView 维护四个核心字段：

1. **m_ids**：当前系统中所有未提交事务的 ID
    
2. **min_trx_id**：未提交事务中最小 ID
    
3. **max_trx_id**：下一个将要分配的事务 ID
    
4. **creator_trx_id**：当前事务 ID
    

---

## ⭐ MVCC 判断版本可见性的规则（核心！）

当一个事务想读取某行数据，需要判断 **该版本是否对当前事务可见**：

假设某行有多个版本（undo 保留旧版本）：

可见性判断：

1. **版本的事务 ID < ReadView.min_trx_id**  
    → 说明版本的事务已经提交 → 可见
    
2. **版本的事务 ID > ReadView.max_trx_id**  
    → 是未来版本 → 不可见
    
3. **版本事务 ID 在 m_ids 中（未提交）**  
    → 不可见
    
4. **否则可见**
    

如果当前版本不可见，则沿着 undo log 找旧版本，一直找到可见的为止。

---

# 🔥 一句话总结（面试官很喜欢）

> MVCC 依赖 Undo Log 和 ReadView，通过“保存历史版本 + 可见性判断”，让普通 select 不加锁，实现高并发读。

# ✅ **Undo Log 是什么？**

一句话定义：

> **Undo Log 是 InnoDB 为了实现回滚和 MVCC 而记录的“旧版本数据”。**

也就是说：

- 当你对一行做 `UPDATE`、`DELETE` 时
    
- InnoDB 会把“修改前的数据”写入 Undo Log
    
- 万一失败了可以回滚
    
- 其他事务也可以用 Undo Log 看到“历史版本”（MVCC）
    

---

# 🔥 **Undo Log 的两大作用（必须会）**

## **1）实现事务的原子性（Atomicity）**

一个事务失败时，需要回滚到修改前的状态。

举例：

```sql
update user set age = 20 where id = 1
```

执行流程：

1）先把原来的数据 `{age = 10}` 写入 Undo Log  
2）然后更新行记录为 `{age = 20}`  
3）如果事务 rollback，就用 Undo Log 把旧值写回去

这就是原子性。

---

## **2）实现 MVCC（多版本并发控制）**

当一个事务在读数据时，如果另一个事务正在修改，那怎么办？

答案：  
→ “读之前的版本”（快照读）

Undo Log 保存了历史版本，比如：

```
版本3：age=18
版本2：age=15
版本1：age=10
```

查询时可以沿着 undo 链找到“读视图可见”的版本。

---

# 🟩 Undo Log 有几种类型？（面试加分）

## **1）Insert Undo Log**

- 记录 `INSERT` 操作
    
- 因为新插入的数据若回滚，是直接删除，所以不影响 MVCC
    
- 事务提交后 → **可直接丢弃**
    

## **2）Update Undo Log（重点）**

- 记录 `UPDATE`、`DELETE` 前的旧数据
    
- 需要提供历史版本给 MVCC 快照读使用
    
- 事务提交后 → **不能立即删除**（因为 MVCC 可能还需要）
    

---

# 🔶 Undo Log 为什么一直增长？需要什么配套机制？

因为：

- 每次更新都会产生 Undo Log
    
- 但是其他事务快照可能还依赖这些“旧版本”
    

所以 Undo Log **不能立刻删除**！

因此需要专门的“清理机制”——**Purge 线程**。

---

# 🔥 Undo Log 的搭档：还有哪些日志一起工作？

Undo Log 只是 MySQL 日志机制的一部分  
完整的 3 种日志体系是：

---

# 🟥 **1）Undo Log（回滚日志）**

作用：

- 回滚事务
    
- MVCC 旧版本
    

---

# 🟧 **2）Redo Log（重做日志） — 实现持久性（D）**

Undo 是记录旧值  
Redo 是记录新值

作用：

- 事务提交之后，先写 redo log
    
- 崩溃恢复时根据 redo log 重放操作
    

---

# 🟨 **3）Binlog（归档日志） — 主从复制 / 审计**

作用：

- 用于同步到从库（主从复制）
    
- 用于恢复到任意时间点（point-in-time recovery）
    
- 记录的是“逻辑日志”——执行了什么 SQL
    

---

# 🔵 面试官超级喜欢你这样回答：

> MySQL 有 Undo Log、Redo Log、Binlog 三类日志，各自负责不同事务阶段：
> 
> - Undo Log 保证“原子性”，提供“旧版本”实现 MVCC。
>     
> - Redo Log 保证“持久性”，崩溃恢复。
>     
> - Binlog 用于“主从复制”和“归档”。
>     

---

# 🟣 Undo Log 为什么需要回收（purge）？

因为 Undo Log 会不断增长  
但不能立刻删除（MVCC 可能还需要旧版本）

回收条件：

1. 使用 Undo Log 的 ReadView 都已经无效
    
2. 事务已经提交
    

Purge 线程会自动回收。

---

# 🟢 一句话总结（背就完事了）

> Undo Log 保存“旧数据”，用于回滚和 MVCC；  
> Redo Log 保存“新数据”，用于崩溃恢复；  
> Binlog 记录 SQL，用于主从复制和审计；  
> Undo Log 需要 purge 回收，否则会无限增长。

# ✅ **1️⃣ TCP 和 UDP 的区别？**

## ⭐ 标准回答

|对比项|TCP|UDP|
|---|---|---|
|是否连接|面向连接（三次握手）|无连接|
|是否可靠|可靠传输（重传、拥塞控制）|不可靠，不保证顺序|
|速度|慢|快|
|传输方式|字节流|数据报|
|开销|大|小（无状态）|
|场景|文件传输、HTTP、邮件|视频、语音、直播、DNS|

## ⭐ 一句话总结

> TCP 可靠但慢，UDP 快但不可靠。

---

# ✅ **2️⃣ TCP 是如何保证可靠性的？**

**TCP 五大机制：**

1. **序列号（保证顺序）**
    
2. **确认应答 ACK（确保送达）**
    
3. **超时重传（丢了就重发）**
    
4. **滑动窗口（流量控制）**
    
5. **拥塞控制（慢启动、拥塞避免）**
    

## ⭐ 一句话总结

> TCP 通过序号 + ACK + 重传 + 滑动窗口 + 拥塞控制实现可靠传输。

---

# ✅ **3️⃣ HTTP 的 keep-alive 是怎么保持长连接的？**

## ⭐ 标准回答

- HTTP/1.0 每次请求都需要一次 TCP 连接（短连接）
    
- HTTP/1.1 默认 **开启 keep-alive**
    

**keep-alive 的原理：**

> 一个 TCP 连接可复用多个 HTTP 请求，不必重复建立和断开 TCP。

服务器不会马上关闭连接，而是保持一段时间（timeout）。

优势：

- 减少三次握手成本
    
- 提高 QPS 和吞吐量
    
- 提升性能
    

---

# ✅ **4️⃣ RESTful 风格接口是什么？**

## ⭐ 标准回答

RESTful 是一种接口设计风格，核心思想：

1. **URL 代表资源**
    
    ```
    GET /users/1
    POST /orders
    DELETE /products/10
    ```
    
2. **用 HTTP 方法表达行为**
    
    - GET：查
        
    - POST：增
        
    - PUT：改
        
    - DELETE：删
        
3. **用状态码表达结果**
    
    - 200 成功
        
    - 201 Created
        
    - 400/404/500 错误
        
4. **无状态**
    
    - 每次请求包含所有信息
        

## ⭐ 一句话总结

> RESTful 用 URL 表资源，用 HTTP 方法表行为，简洁、规范、无状态。

# ✅ **5️⃣ TCP 三次握手每一步做什么？为什么不是两次？**

## ⭐ 三次握手步骤

1. **客户端 → 服务端：SYN**  
    “我想连接你”
    
2. **服务端 → 客户端：SYN + ACK**  
    “我同意连接，并且我也能正常发送”
    
3. **客户端 → 服务端：ACK**  
    “我确认回复，我能正常收发”
    

---

## ⭐ 为什么不是两次？

因为必须双方都确认“发送”和“接收”能力正常。

两次握手无法确认客户端的接收能力，会出现 **历史连接问题**（老旧 SYN 包建立连接）。

一句话总结：

> 三次握手是为了防止历史连接问题，并确保双方都有接收和发送能力。

---

# ✅ **6️⃣ HTTP 和 HTTPS 的区别？如何实现加密？**

## ⭐ 区别

HTTP：

- 明文传输
    
- 无加密
    
- 劫持风险高
    

HTTPS：

- HTTP + TLS/SSL
    
- 有加密、认证、完整性校验
    

## ⭐ HTTPS 如何加密？

通过 **非对称加密 + 对称加密结合**：

1. 客户端拿到证书（RSA 公钥）
    
2. 使用公钥加密“对称密钥”
    
3. 后续通信用对称密钥（AES）加密
    

优势：  
速度快（对称） + 安全（非对称）

# ✅ **7️⃣ Token 和 Session 的区别？分布式系统如何管理 Session？**

## ⭐ Token vs Session

|项目|Session|Token|
|---|---|---|
|存储|服务端（内存/Redis）|客户端（jwt）|
|状态|有状态|无状态|
|扩展性|不好（需要会话共享）|好（不用共享）|
|安全|不可篡改（服务端存储）|可签名防篡改|

## ⭐ 分布式 Session 解决方案：

1. **Session 共享：存 Redis**（主流）
    
2. **Session Sticky（会话粘滞）**
    
3. **JWT（官网推荐，无需共享）**

# ✅ **8️⃣ Redis 的缓存雪崩、击穿、穿透**

## ⭐ 1）缓存穿透（key 永远查不到）

- 恶意请求不存在的 key
    
- Redis 查不到 → DB 查不到 → 压力很大
    

**解决：**

- 布隆过滤器（BloomFilter）
    
- 对不存在的数据缓存空值（短过期）
    

---

## ⭐ 2）缓存击穿（热点 key 过期了）

- 大量请求打到一个热点 key
    
- key 过期，瞬间打到数据库
    

**解决：**

- 热点 key 永不过期
    
- 分布式锁控制只有一个线程回源 DB
    

---

## ⭐ 3）缓存雪崩（大量 key 同时过期）

- 大量缓存同时失效
    
- 请求全部打到数据库 → 系统崩溃
    

**解决：**

- 过期时间添加随机值，避免同时失效
    
- 多级缓存（Redis + 本地缓存）
    
- 限流/降级
    

---

# 🎯 一句话总结

> 穿透：查不存在 → BloomFilter  
> 击穿：热点 key 失效 → 加锁  
> 雪崩：大量 key 同时过期 → 随机过期时间 + 多级缓存

结构非常清晰，一说就稳。

---

# ✅ 1️⃣ 缓存穿透（Cache Penetration）

## 🟥 **问题本质：查“根本不存在的数据”**

例如攻击者不停请求：

```
/user?id=-1
/user?id=99999999（库里没有）
```

特点：

- Redis 查不到
    
- DB 也查不到
    
- 流量直接打到数据库 → 数据库崩
    

---

# 🟩 **如何避免（解决方案）**

## ⭐ 方案 1：布隆过滤器（Bloom Filter）

提前将合法的 key（比如 id）加入布隆过滤器：

- 请求过来 → 先查布隆过滤器
    
- 判断 key 是否可能存在
    
- 大概率不存在的 key 直接拦截，不进数据库
    

**大规模系统使用最多（京东、美团）**

---

## ⭐ 方案 2：缓存空对象（empty-value）

如果某个 key 不存在：

```json
user:null
```

并给一个较短过期时间（如 2 分钟）

- 下次请求直接返回空
    
- 不会再访问数据库
    

但是要注意防止大量空值占内存。

---

## ⭐ 方案 3：参数校验

在进入系统前就拦截（如 id < 0 直接拒绝）

---

# 🎯 **总结一句话（面试可直接说）**

> 穿透是“不存在的数据一直查 DB”，通过布隆过滤器或缓存空对象解决。

---

# ✅ 2️⃣ 缓存击穿（Cache Breakdown）

## 🟥 **问题本质：热点 key 突然失效**

例如：

某个 **高并发热点数据**（如明星商品、热点新闻）在某一个时间点刚好过期：

- 大量请求同时到来
    
- 发现 Redis 没有
    
- 全部打到 DB → 数据库压力瞬间暴涨
    

---

# 🟩 **如何避免（解决方案）**

## ⭐ 方案 1：热点数据永不过期

把热门 key 设置为：

- **不设置过期时间**
    
- 或后台异步刷新
    

适用场景：热点非常稳定（如配置信息）

---

## ⭐ 方案 2：互斥锁（分布式锁防止同时回源）✔ 推荐

流程：

1）第一个请求发现缓存失效  
2）加锁（Redis setnx）  
3）其他请求等待  
4）第一个请求从 DB 查，写回缓存  
5）释放锁

伪代码：

```java
if (cache == null) {
    if (setnx(lock_key)) {
        dbQueryAndSetCache();
        delete(lock_key);
    } else {
        sleep(50ms);
        retry();
    }
}
```

**保证同一时刻只有 1 个请求能回源 DB**

---

## ⭐ 方案 3：逻辑过期（缓存中放一个“过期字段”）

缓存存储：

```json
{
  "data": {...},
  "expire": 1680000000
}
```

过期时：

- 返回旧数据
    
- 后台线程异步更新缓存
    

**不会阻塞用户请求，用户永远有数据可用。**

---

# 🎯 **面试一句话总结**

> 击穿是“热点 key 到期，一瞬间大量请求打 DB”，可以通过互斥锁、逻辑过期、热点数据永不过期解决。

---

# ✅ 3️⃣ 缓存雪崩（Cache Avalanche）

## 🟥 **问题本质：大量 key 在同一时间同时失效**

例如：

- 大量 key 统一设置 TTL = 2 小时
    
- 恰好 2 小时后全部过期
    
- 所有请求命中空缓存 → DB 压力暴增 → DB 直接被打死
    

是 **大范围问题**，危害比击穿更大。

---

# 🟩 **如何避免（解决方案）**

## ⭐ 方案 1：过期时间加随机值（最常用）

不要让大量 key 同时过期。

例如：

```java
TTL = 3600 + random(0, 600)  // 加10分钟随机值
```

是最简单也最有效的方式。

---

## ⭐ 方案 2：缓存预热

上线前或系统启动后提前把关键数据加载到缓存。

---

## ⭐ 方案 3：多级缓存（本地缓存 + Redis）

常见架构：

- 一级缓存：Guava Cache / Caffeine
    
- 二级缓存：Redis
    
- 三级：数据库
    

即使 Redis 崩，也可以从本地缓存顶住短时间流量。

---

## ⭐ 方案 4：限流 + 降级

比如：

- Redis 失效时开启限流（如 Sentinel、Nginx）
    
- 数据库压力过高时降级（返回默认页、降级提示）
    

---

# 🎯 面试一句话总结

> 雪崩是“大量 key 同时过期 → 流量击穿数据库”，可通过随机过期、多级缓存、限流降级解决。

---

# 🎉 最终可背版本（10 秒说完）

> **穿透**：查不存在的数据 → **布隆过滤器 / 缓存空值**  
> **击穿**：某热点 key 失效 → **互斥锁 / 永不过期 / 逻辑过期**  
> **雪崩**：大量 key 同时失效 → **随机过期时间 / 多级缓存 / 限流降级**
